# LightSpec — Product Requirements Document

**Date:** 2026-05-04  
**Status:** Draft

---

## Problem

OpenSpec is a solid spec-driven workflow, but it has two friction points:

1. **Token cost** — artifacts are verbose (multiple long .md files per change, bloated skill prompts with extensive scaffolding instructions). On large features or long sessions this adds up fast.
2. **Regression is an afterthought** — the verify step checks that tasks were completed and specs were followed, but it does not systematically ask "what did this change break?" That's the question that matters most on brownfield projects.

---

## Goal

LightSpec is a lean, skill-based spec workflow for Claude Code that:

- Cuts token usage vs OpenSpec by keeping specs compact and skill prompts minimal
- Makes regression prevention a first-class concern, not a footnote
- Uses the same skill/slash-command mechanism as OpenSpec (SKILL.md files in `.claude/skills/`)
- Requires no external CLI — pure skills, no npm install

---

## Non-goals

- Feature parity with OpenSpec (no bulk-archive, onboard, sync, dashboard, multi-schema, etc.)
- Multi-tool / multi-agent support (Claude Code only, at least v1)
- A GUI or web dashboard

---

## Core Concepts

### One compact spec file per change

Instead of `proposal.md` + `design.md` + `specs/` + `tasks.md`, LightSpec uses a **single `spec.md`** per change with fixed, short sections:

```
lightspec/changes/<name>/spec.md
```

Sections (all kept short — bullet lists, not prose):

| Section | Purpose | Max length |
|---|---|---|
| `## What` | One-sentence description of the change | 1–2 lines |
| `## Why` | Motivation | 2–4 bullets |
| `## Scope` | Files/areas touched | bullet list |
| `## Deltas` | Behavioral requirements + scenarios for this change | see below |
| `## Risks` | What could regress | bullet list — **required** |
| `## Tasks` | Checkbox implementation steps | bullet list |

No YAML frontmatter. No templates. No scaffolding boilerplate copied into the file.

### Delta specs

Borrowed from OpenSpec — the `## Deltas` section captures the behavioral contract for this specific change. Not the whole system, just what's new or different.

Format (inline, no separate files):

```markdown
## Deltas

### Requirement: <name>
<one-line description of what must now be true>

#### Scenario: <scenario name>
- Given: <precondition>
- When: <action>
- Then: <expected outcome>
```

Keeping deltas inline with the rest of `spec.md` avoids the multi-file overhead of OpenSpec's `specs/` directory while preserving the behavioral contract that makes verify meaningful. A typical change needs 1–3 requirements and 2–5 scenarios — keep them tight.

`/ls:verify` uses these deltas as its correctness checklist (does the implementation satisfy each scenario?), separate from the regression/risk checklist.

### Regression-first verify

The `/ls:verify` skill is the differentiator. It does not just check task completion — it actively hunts regressions:

1. Reads the `## Risks` section from the spec
2. Reads the `## Scope` section to know which files changed
3. For each risk + each changed file, searches the codebase for callers, dependents, and related tests
4. Produces a **Regression Report** with pass/fail/unknown per risk item
5. Flags any changed function signature, removed export, or changed return shape, and traces its callers

## Commands (Slash Skills)

| Command | What it does |
|---|---|
| `/ls:propose <name>` | Creates `spec.md` scaffold, asks clarifying questions only if needed, populates all sections |
| `/ls:implement` | Reads `spec.md`, works through `## Tasks` checkboxes, marks each done |
| `/ls:verify` | Regression-first verification — see detail below |
| `/ls:done` | Archives the change (moves to `lightspec/archive/`) |

### `/ls:propose`

**Goal:** produce a filled `spec.md` in one pass with minimal back-and-forth.

Behaviour:
- Accepts a change name or description (derive kebab-case name from description)
- Populates all five sections using codebase context
- `## Risks` is generated by scanning what the scope touches — not skipped, not left blank
- If scope is unclear, asks one focused question only
- No external CLI calls

Token budget: the entire `spec.md` output should typically be under 400 tokens.

### `/ls:implement`

**Goal:** work through tasks, stay focused, pause cleanly on blockers.

Behaviour:
- Reads `spec.md` only (not a stack of files)
- Implements tasks in order, ticking checkboxes as it goes
- Pauses and describes the blocker if a task is ambiguous or reveals a design conflict
- Does not pad output with progress theatre — just shows current task and result

### `/ls:verify`

**Goal:** catch regressions before the change is archived.

This is the centrepiece skill. Steps:

1. **Load spec** — read `spec.md`, extract `## Deltas`, `## Risks`, `## Scope`, `## Tasks`

2. **Task completeness check** — count unchecked tasks. If any exist, report CRITICAL and stop (no point verifying code that isn't done).

3. **Changed-file map** — `git diff --name-only` (or read `## Scope`) to get the actual changed files.

4. **Caller / dependent scan** — for every function, class, or exported symbol touched in those files:
   - `grep` for its name across the project
   - Identify callers outside the changed files
   - Check if the call-site still compiles (type signatures, required args, return type usage)
   - Flag anything that looks broken or suspicious

5. **Delta correctness check** — iterate `## Deltas` scenarios. For each scenario:
   - Search the implementation for code that satisfies the Given/When/Then
   - Check for a corresponding test (or note if untested)
   - Mark: ✅ Satisfied / ⚠️ Untested / ❌ Not implemented

6. **Risk checklist** — iterate the `## Risks` bullets from the spec. For each:
   - Search for evidence the risk was addressed (a test, a guard, a comment)
   - Mark: ✅ Mitigated / ⚠️ Unverified / ❌ Likely broken

7. **Test coverage check** — for each changed file, check whether a test file exists and whether it exercises the changed paths. Flag gaps.

7. **Regression Report output:**

```
## Regression Report: <change-name>

### Task Completeness
X/Y tasks complete [PASS / CRITICAL]

### Changed Files
- src/foo.ts
- src/bar.ts

### Caller Impact
| Symbol | Callers Found | Status |
|---|---|---|
| doThing() | src/baz.ts:42, src/qux.ts:11 | ✅ OK |
| getConfig() | src/app.ts:7 | ⚠️ Return shape changed — verify caller |

### Delta Correctness
| Scenario | Status | Notes |
|---|---|---|
| User sees error on invalid input | ✅ Satisfied | src/validate.ts:34, test exists |
| Empty state renders correctly | ⚠️ Untested | code present, no test |

### Risk Checklist
- ✅ Existing auth flow unaffected (verified: no changes to auth/)
- ⚠️ API response shape — no test coverage found
- ❌ Cache invalidation — logic removed in src/cache.ts:88

### Test Coverage
- src/foo.ts → tests/foo.test.ts ✅
- src/bar.ts → no test file found ⚠️

### Verdict
CRITICAL: 1 | WARNING: 2 | OK: 2
Not ready to archive. Fix critical items first.
```

### `/ls:done`

Moves `lightspec/changes/<name>/` to `lightspec/archive/<date>-<name>/`. No output beyond confirming the move.

## Token Efficiency Design Decisions

- **Single file per change** — no loading 4–5 separate artifacts per command
- **No templates pasted into skills** — skills describe intent and heuristics, not scaffolding text for the AI to copy
- **Short skill prompts** — each SKILL.md targets ~200–400 lines (OpenSpec skills run 100–300 lines but load multiple external CLI outputs on top)
- **No CLI dependency** — no `openspec instructions`, `openspec status --json` round-trips; the spec file is the single source of truth
- **`/ls:verify` searches lazily** — scans only changed-file callers, not the entire codebase

## Implementation Plan

### Phase 1 — Pure skills (no CLI)

Deliver four SKILL.md files installable via:

```bash
mkdir -p .claude/skills/ls-propose .claude/skills/ls-implement .claude/skills/ls-verify .claude/skills/ls-done
# copy SKILL.md files into each
```

No build step, no npm publish, no external tooling.

### Phase 2 — Optional init script (TBD)

A small script (language TBD — Node.js, Python, or shell) that:
- Creates the `lightspec/` directory structure
- Copies skills into `.claude/skills/`
- Optionally wires up a `CLAUDE.md` mention

No requirement for Phase 1.

---

## Success Metrics

- `/ls:verify` catches at least one regression risk per non-trivial change
- Token usage per full propose→implement→verify cycle measurably lower than OpenSpec equivalent
- No external CLI required to use v1

---

## Open Questions

1. **Language for Phase 2 CLI** — Node.js (matches OpenSpec), Python (simpler), or pure shell. Leaning toward a minimal shell script or no CLI at all if skills prove sufficient.
2. **Multi-change support** — should `/ls:implement` auto-detect the active change or always require a name? (Lean: auto-detect if only one active change exists)
3. **CLAUDE.md integration** — should `ls-propose` emit a reminder to run `/ls:verify` before archiving, written into the project's CLAUDE.md?
